# 기여 가이드 (Contributing Guide)

이 프로젝트에 기여해주셔서 감사합니다. 아래 가이드라인을 따라주시면 원활한 협업에 큰 도움이 됩니다.

## 1. Issue 생성

버그 리포트, 기능 제안 등 모든 기여는 Issue를 통해 시작하는 것을 권장합니다.

- **버그 리포트:** 재현 가능한 상세한 설명과 함께 작성해주세요.
- **기능 제안:** 어떤 문제를 해결하고 싶은지, 제안하는 기능이 어떤 가치를 제공하는지 명확히 설명해주세요.
- **라벨(Label):** `버그`, `기능`, `문서` 등 적절한 라벨을 붙여주세요.

## 2. Pull Request (PR) 워크플로우

1.  이 저장소를 Fork합니다.
2.  새로운 기능이나 수정을 위한 브랜치를 생성합니다. (`git checkout -b feature/새-기능-이름`)
3.  코드를 수정하거나 문서를 작성합니다.
4.  변경 사항을 커밋합니다. (아래 '커밋 메시지 컨벤션' 참고)
5.  Fork한 저장소에 Push합니다.
6.  원본 저장소로 Pull Request를 생성합니다.
7.  PR 템플릿에 따라 변경 사항을 상세히 설명합니다.

## 3. 번역 기여

본 문서는 다국어(한국어, 영어)를 지원합니다.

- **번역 추가/수정:** `docs/ko` 또는 `docs/en` 디렉토리의 파일을 수정합니다.
- **일관성:** 두 언어의 문서 구조와 내용이 최대한 일치하도록 유지합니다.
- **PR 생성:** 번역 완료 후, PR 제목에 `[번역]` 또는 `[i18n]` 말머리를 추가해주세요.

## 4. 커밋 메시지 컨벤션

커밋 메시지는 변경 사항을 명확히 알 수 있도록 아래 형식에 따라 작성합니다.

```
타입: 제목

(선택) 본문

(선택) 꼬리말
```

- **타입(Type):**
    - `feat`: 새로운 기능 추가
    - `fix`: 버그 수정
    - `docs`: 문서 변경
    - `style`: 코드 스타일 변경 (포맷팅 등)
    - `refactor`: 리팩토링
    - `test`: 테스트 코드 추가/수정

**예시:**
```
docs: 기여 가이드라인에 커밋 메시지 컨벤션 추가

프로젝트의 커밋 히스토리 일관성을 위해 Angular 스타일의 커밋 메시지 규칙을 추가함.
```
